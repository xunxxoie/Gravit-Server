# 데이터베이스 규칙

## Flyway 마이그레이션

- 파일 경로: `src/main/resources/db/migration/`
- 파일명 형식: `V{N}__{description}.sql` (언더스코어 2개, description은 snake_case)
- 적용된 마이그레이션 파일은 절대 수정하거나 삭제하지 마라 (Flyway checksum 검증 실패)
- 스키마 변경이 필요하면 새 버전 파일을 추가하라
- 각 SQL 문 앞에 주석으로 목적을 설명하라

## SQL 스타일

- 테이블명, 컬럼명: snake_case (e.g., `wrong_answered_note`, `profile_img_number`)
- PK: `id BIGINT GENERATED BY DEFAULT AS IDENTITY`를 사용하라
- FK 컬럼: `{참조테이블}_id` 형식으로 네이밍하라 (e.g., `problem_id`, `user_id`)
- timestamp: `TIMESTAMP(6)`을 사용하라
- enum: `VARCHAR(255)` + `CHECK` 제약 조건을 사용하라
- JSON: `JSONB` 타입을 사용하라
- 인덱스: `CREATE INDEX IF NOT EXISTS ix_{테이블}_{용도}` 형식으로 네이밍하라

## JPA Entity ↔ DB 매핑

- Entity 클래스명과 테이블명이 다를 경우 `@Table(name = "...")`을 명시하라
- 컬럼명은 `@Column(name = "snake_case")`으로 명시하라
- enum 매핑은 `@Enumerated(EnumType.STRING)`을 사용하라. ORDINAL을 사용하지 마라
- ID 생성은 `@GeneratedValue(strategy = GenerationType.IDENTITY)`를 사용하라
- Soft delete가 필요하면 `@SQLRestriction` + `@SQLDelete` 패턴을 사용하라

## Repository

- JPA Repository 인터페이스는 `{domain}/repository/` 패키지에 위치시켜라
- 복잡한 쿼리는 `repository/custom/` 또는 `repository/sql/`로 분리하라
- Projection은 `dto/response/`의 record로 직접 반환하라
